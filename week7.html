Week 7: Tablet Optimization & Cashier Features

April 6-11 2026

10 min read

Bug Fixes and Live Updates

On Monday (April 6), I fixed several critical bugs including a database field mismatch (fist_name vs first_name in UserSeeder) and a default sex value being saved when the field no longer existed. I removed redundant "no orders found" messages and implemented live status indicator updates so order cards refresh instantly when item status changes without requiring a page reload.

I also fixed an item duplication bug where the code was iterating through all order items instead of filtering by order type. I grouped items by type in the order details panel, added a white background to the sticky table header, and made the status badge update live when order items are marked as "preparing".

Unexpected Tablet Deployment

On Tuesday (April 7), Sir George surprised us by providing Fire 7 tablets (600x1024) that Cynthia Curry waiters would actually use in production. This forced us to immediately shift to responsive design after we had been focused on laptop screen sizes. We used one teammate's laptop as a temporary server to test our currently developed system on the actual tablets. The smaller resolution meant we had to reorganize layouts, resize buttons for easier touch interaction, and carefully manage information density to avoid overwhelming the screen. Many UI elements that worked fine on a laptop felt cramped on the tablet, requiring us to prioritize what information should be visible at each step.

In the afternoon, I used my own laptop to test my changes and see how they would look and function on the tablet. I added color-coded table numbers for each room and an "All Rooms" filter option as suggested by teammates. I implemented live updates on the kitchen orders page using polling (every 2 seconds) after WebSocket attempts failed. I also fixed the PIN modal to accept only numeric input and allowed users to press Enter to submit instead of clicking confirm.

Cashier Features and Spice Level

On Wednesday (April 8), our mentor showed us another system to evaluate. We did a quality assurance test on the system given to us, found several things that could be better, and reviewed the source code to understand best practices. After completing the QA review, we went back to developing the POS. I began building the cashier's orders page, testing it on the actual target device to ensure it fit properly.

Thursday (April 9) focused on refining the UI to match the billout page style, and in the afternoon, I copied and adapted the kitchen orders logic for the cashier's needs.

On Friday (April 10), I added a nullable spice_level column to the order_items table (for curry items only) and modified the Order Controller to save and display this data. I implemented the cancel/void feature with a success modal and added 2-second polling to auto-refresh the orders page.

Payment Processing

On Saturday (April 11), I implemented the billout page backend with dynamic database-driven information. I disabled the numpad until the cash button is clicked, disabled GCash when cash is selected, and implemented discount functionality for both whole orders and individual items.

The system now automatically calculates change when cash is entered. I created migrations and models to persist billout data and fixed several bugs to polish the feature.

Key Learnings from Week 7

  • Responsive design is critical when deploying to multiple device types with different screen sizes
  • Data synchronization between multiple client interfaces requires thoughtful planning and implementation
  • Polling can be a practical short-term solution for real-time updates, though WebSockets are more efficient long-term
  • Consistent UI/UX across related pages improves usability and user familiarity
  • Understanding end-user requirements early prevents rework and ensures better product-market fit

Week 7 Reflection

This was an intense week filled with unexpected changes and learnings. The surprise tablet deployment forced us to pivot our design approach, teaching me the importance of gathering complete requirements upfront. I also learned valuable lessons about real-time data synchronization and made practical tradeoffs between ideal solutions and workable implementations. By week's end, the POS system was becoming more robust, with major features for the kitchen, orders, and cashier workflows working well. The team collaboration on UI improvements and the willingness to adapt to changing requirements demonstrated the importance of agility in software development.
